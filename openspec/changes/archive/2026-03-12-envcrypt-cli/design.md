## Context

現在、`.env` ファイルはチーム間での共有にSlackやメール等の非セキュアな手段が使われがちで、Gitにはコミットできない。envcrypt は `.env` を暗号化してGitで安全に管理できるようにするGo製CLIツール。マスターキーはクラウド(GCP/AWS)のSecret Managerに保管し、ローカルに鍵を持たない設計とする。

## Goals / Non-Goals

**Goals:**
- シンプルな `encrypt` / `decrypt` の2コマンドで完結するCLI
- AES-256-GCM による認証付き暗号化で機密性と完全性を保証
- ファイルごとにランダムなデータキーを生成し、マスターキーでラップするエンベロープ暗号化
- GCP Secret Manager と AWS Secrets Manager の両方をサポート
- `dotenv.yaml` による宣言的な設定管理
- 独自バイナリフォーマット(ENVCヘッダー)による暗号化ファイル

**Non-Goals:**
- Google Cloud KMS / AWS KMS との直接統合（将来拡張）
- 環境別のキー管理（将来拡張）
- Git pre-commit hook の自動設定
- CI/CD パイプラインとの統合
- 複数ファイルの一括暗号化（将来拡張）
- Secret Manager のバージョンピンニング（将来拡張）

## Decisions

### 1. CLIフレームワーク: cobra + viper
**選択理由**: Go CLIのデファクトスタンダード。サブコマンド構造、フラグ管理、設定ファイル読み込みが統合されている。
**代替案**: urfave/cli — 軽量だが、viper との統合が弱い。

### 2. エンベロープ暗号化パターン
**選択理由**: ファイルごとにランダムなデータキーを生成し、マスターキーでラップすることで、1つのファイルが漏洩しても他のファイルの安全性に影響しない。マスターキーのローテーションも容易。
**代替案**: マスターキーで直接暗号化 — キーローテーション時に全ファイルの再暗号化が必要になる。

### 3. AES-256-GCM
**選択理由**: Go標準ライブラリで利用可能。AEAD（認証付き暗号化）により改ざん検知が組み込まれている。
**代替案**: ChaCha20-Poly1305 — パフォーマンス面で有利だが、AESの方が広く知られている。

### 4. 独自バイナリフォーマット
**選択理由**: ヘッダーにマジックバイト(ENVC)とバージョンを含めることで、ファイルの識別とフォーマットの後方互換性を確保。
**代替案**: JSON/Base64 — 人間が読める利点があるが、ファイルサイズが増大する。

### 5. Secret Provider の抽象化
**選択理由**: GCP と AWS を共通インターフェースで抽象化し、`dotenv.yaml` の `cloud` フィールドで切り替え。新しいプロバイダの追加も容易。
**代替案**: GCP固定 — シンプルだがAWSユーザーを排除する。

## Risks / Trade-offs

- **[Secret Manager への依存]** → オフライン環境では使えない。ミティゲーション: エラーメッセージで認証状態を明確に通知。
- **[独自バイナリフォーマット]** → 他ツールとの互換性がない。ミティゲーション: バージョンフィールドにより将来のフォーマット変更に対応可能。
- **[データキーのラップにAES-KWではなくAES-GCMを使用]** → 標準的なKey Wrappingアルゴリズムではない。ミティゲーション: AES-GCMのAEAD特性で同等のセキュリティを確保。
- **[設定ファイルの誤設定]** → 間違ったプロジェクトIDやシークレットIDを指定するリスク。ミティゲーション: encrypt/decrypt実行時にSecret Managerへの接続を検証。
